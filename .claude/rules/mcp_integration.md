# MCP Integration

MCP (Model Context Protocol) connections are **mandatory** for this framework. Two connections are required:

1. **Issue tracker MCP** (Linear, ClickUp, Jira) — reading features, checking dependencies, updating status
2. **Git provider MCP** (GitHub, GitLab, Bitbucket) — creating branches, PRs, reading PR review comments

The live connections are defined in `.mcp.json` (project scope, committed to the repo). Set them up with `claude mcp add --scope project <name> ...` and verify with `claude mcp list` or `/mcp` inside a session. In unattended cloud surfaces (Claude Code on the web / Routines), the same project-scoped `.mcp.json` is used, and the servers' credentials are supplied by the environment rather than a local machine.

If either MCP is unavailable when an agent starts, the agent must **stop immediately** with a clear error:

> "⛔ **MCP ERROR:** {Issue tracker | Git provider} MCP is not available. This framework requires both MCP connections to operate. Run `claude mcp list` to check, and see `docs/DEVELOPMENT.md` for setup instructions."

## 0. Pre-Requisite: `project_state.json`

All commands (except `/init-project`) require `.claude/project_state.json` to exist. This file is generated by `/init-project` and committed to the repo.

**Before reading any MCP configuration, check:**

1. Does `.claude/project_state.json` exist?
2. If **NO**: **STOP.** "⛔ `.claude/project_state.json` not found. Run `/init-project` first to set up MCP configuration."
3. If **YES**: Read and parse the file. Proceed.

## 1. Configuration: `.claude/project_state.json`

Generated by `/init-project`. Committed to the repo. Read by every agent. This is the framework's own state file (status map + feature registry). It is distinct from `.mcp.json`, which is Claude Code's live MCP server configuration.

### 1.1 Schema

```json
{
  "issue_tracker": {
    "provider": "<linear | clickup | jira>",
    "mcp_server": "<MCP server name as registered in .mcp.json>",
    "project": { "id": "<external project ID>", "name": "<project name>" },
    "team": { "id": "<external team ID>", "name": "<team name>" }
  },
  "git_provider": {
    "provider": "<github | gitlab | bitbucket>",
    "repository": "<owner/repo-name>"
  },
  "status_mapping": {
    "todo": "<tracker status name>",
    "planning": "<tracker status name>",
    "plan_review": "<tracker status name>",
    "ready_for_build": "<tracker status name>",
    "in_progress": "<tracker status name>",
    "in_review": "<tracker status name>",
    "done": "<tracker status name>"
  },
  "reverse_status_mapping": {
    "<tracker status name>": "<framework status or null>"
  },
  "features": {
    "<FEATURE_ID>": {
      "external_id": "<UUID in tracker>",
      "external_identifier": "<human-readable key>"
    }
  }
}
```

### 1.2 Status Mapping

The framework defines these logical statuses. During `/init-project`, they are mapped to the tracker's actual status names. If the tracker has fewer statuses, multiple framework statuses map to the same tracker status (this is fine).

| Framework Status | Purpose | Set by |
| --- | --- | --- |
| `todo` | Not started | Default |
| `planning` | `/plan-feature` agent is generating the plan | Plan agent |
| `plan_review` | Plan PR created, awaiting human review | Plan agent |
| `ready_for_build` | Plan approved, ready for implementation | Human (in tracker) |
| `in_progress` | `/build-feature` agent is implementing | Build agent |
| `in_review` | Implementation PR ready, awaiting human review | Build agent |
| `done` | PR merged to main | CI pipeline (automated) |

**Minimum viable mapping:** The tracker must have at least statuses that can represent "not started", "in progress", "in review", and "done". Other framework statuses can share these.

## 2. Reading Features from MCP

Agents never read feature details from local files. All feature data comes from the issue tracker MCP:

1. **By Feature ID:** Look up the feature's `external_id` in `project_state.json`, then call the tracker's "get issue" tool.
2. **Extract:** Title, description, acceptance criteria, priority, current status, labels/tags.
3. **If the feature is not found in `project_state.json`:** Stop and report: "Feature {FEATURE_ID} not found in project_state.json. Run `/init-project` to refresh the feature list."

## 3. Dependency Checking

Before starting any work, agents check dependencies:

1. Read `feature_map.md` to find the feature's `depends_on` list.
2. For each dependency, look up its `external_id` in `project_state.json`.
3. Query the tracker MCP for each dependency's current status.
4. Map each status via `reverse_status_mapping` to a framework status.
5. If **any** dependency is not `done`, **stop:** "⛔ **BLOCKED:** {FEATURE_ID} depends on {DEP_ID} which is currently '{status}'. All dependencies must be Done before this feature can proceed."

## 4. Updating Feature Status

When an agent needs to update a feature's status:

1. Look up the feature's `external_id` in `project_state.json`.
2. Look up the target framework status in `status_mapping` to get the tracker's status name.
3. Call the tracker's "update issue" tool with the mapped status name.
4. If the call fails, warn: "MCP status update failed for {FEATURE_ID}: {error}. The workflow continues." **Do not block the workflow on status update failures.**

## 5. Git Provider MCP Operations

### 5.1 Branch and PR Operations

- **Create branch:** Use Git provider MCP or `git` CLI (both are acceptable).
- **Create PR:** Use Git provider MCP. Include feature ID in the PR title.
- **Update PR:** Use Git provider MCP to convert draft → ready, update title, update description, add labels.

### 5.2 Reading PR Review Comments

Used by `/revise-feature` and `/plan-feature` (when re-planning):

1. Identify the PR for the feature branch via Git provider MCP.
2. Fetch all review comments (both inline and general).
3. Parse comments to extract actionable feedback.
4. If no Git provider MCP is available, stop: "Git provider MCP required. Connect it (`claude mcp add`) or paste feedback in the task prompt."

## 6. Graceful Degradation

1. **Status update failures:** Warn but continue. The implementation matters more than the status label.
2. **Feature not in manifest:** Stop — the feature hasn't been imported.
3. **MCP server unavailable at startup:** Stop — agents cannot operate without MCP.
4. **MCP server becomes unavailable mid-operation:** Complete the current step, then warn. Do not block commits or pushes on MCP failures.
5. **`project_state.json` missing:** Stop — run `/init-project` first.
