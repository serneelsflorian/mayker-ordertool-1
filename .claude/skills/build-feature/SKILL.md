name: build-feature
description: Implement one feature from an approved plan. Use when running /build-feature {FEATURE_ID}, building or implementing a planned feature, scaffolding project infrastructure, writing code plus unit/integration tests, generating E2E and UAT specs, running the self-review and refactor gates, and converting the PR to ready for review. Requires an approved plan. Runs autonomously (Claude Code on the web / Routines).
---

# Workflow: Build Feature

**Trigger:** `/build-feature {FEATURE_ID}`
**Run by:** Autonomous session (one per feature) — Claude Code on the web, `claude --remote`, or a local session.

> Delegation: implementation Phases 0S–D run through the `builder` subagent (`.claude/agents/builder.md`), which can read/write code, run tests, and commit. Phase E (self-review) runs through the read-only `reviewer` subagent (`.claude/agents/reviewer.md`), which reports findings the builder then fixes. The test gate is also enforced deterministically by the PreToolUse hook in `.claude/settings.json` on every `git push`.

---

## 0. Load Context

1. Verify `.claude/project_state.json` exists. If not, **STOP:** "Run /init-project first to generate MCP configuration."
2. Read `CLAUDE.md` for project description, tech stack, test configuration, design reference, API references, feature toggles, architecture notes.
3. Read the standards in `.claude/rules/` — in particular `testing_standards.md`, `review_standards.md`, and `refactoring_standards.md` (the universal ones are already in context via CLAUDE.md imports).
4. Read `.claude/project_state.json` for tracker configuration.
5. Read `.claude/feature_map.md` to find this feature's metadata.

---

## 1. MCP Verification

1. Verify issue tracker MCP is available. If not, **STOP** per `mcp_integration.md`.
2. Verify Git provider MCP is available. If not, **STOP** per `mcp_integration.md`.

---

## 2. Dependency Check

1. Read `depends_on` from `feature_map.md`.
2. Query MCP for each dependency's status.
3. If **any** dependency is not `done`, **STOP:** "⛔ BLOCKED: {FEATURE_ID} depends on {DEP_IDS}. All must be Done."

---

## 3. Status & Plan Verification

1. Fetch this feature's current status from MCP.
2. Map via `reverse_status_mapping`. Expected: `ready_for_build`.
   - If `todo` or `planning` or `plan_review`: **STOP.** "Feature is not yet approved for building. Current status: {status}. Run /plan-feature first and get the plan approved."
   - If `in_progress`: This may be a resume. Check if the feature branch has implementation commits. If yes, continue from where it left off. If no, proceed normally.
   - If `in_review` or `done`: **STOP.** "Feature is already in review or done."
3. Check that `.claude/artifacts/{FEATURE_ID}/plan.md` exists (on the feature branch or main).
   - If not found: **STOP.** "No plan found. Run /plan-feature {FEATURE_ID} first."

---

## 4. Branch Setup

1. Checkout the feature branch (created during `/plan-feature`).
2. Pull latest from main: `git fetch origin main` (do NOT auto-rebase — the user triggers rebasing manually when needed).
3. Verify the branch is up to date with the plan.

---

## 5. Update Status → In Progress

Update the feature's status in the tracker to the `in_progress` mapped status.

---

## 6. Read the Plan

Read `.claude/artifacts/{FEATURE_ID}/plan.md`. The plan determines which phases execute:

- **Infrastructure Scaffolding section exists:** Execute Phase 0S (Scaffold) first
- **Frontend Plan section exists and has work:** Execute Phase A (Frontend)
- **Backend Plan section exists and has work:** Execute Phase B (Backend)
- **API Integration Plan section has work:** Execute Phase C (Integration)
- Sections that say "No {x} changes required" or "Not applicable" are **skipped**.

---

## 7. Phase 0S — Infrastructure Scaffolding (scaffold feature only)

> Skip if the plan does not contain an "Infrastructure Scaffolding" section.

This phase runs only for the scaffold feature (the first feature built in the project). It creates the foundational project structure before implementing the feature itself.

1. Read the **Infrastructure Scaffolding** section from `plan.md`.
2. Create project directory structure per the plan.
3. Create Docker configuration:
   - Dockerfile(s) for each service
   - `docker-compose.yml` with all services (app, database, etc.)
   - `.env.example` with all required environment variables (documented with comments)
4. Create test infrastructure:
   - Shared test fixture files (per `CLAUDE.md` Test Configuration)
   - Test directory structure (unit, integration, E2E, UAT directories)
   - E2E framework configuration file
   - UAT directory structure with subdirectories
5. Verify CI pipeline files exist (generated by `/init-project`). If they reference test commands or Docker services that need updating based on the actual project structure, update them.
6. Commit: `chore({FEATURE_ID}): scaffold project infrastructure`

---

## 8. Phase A — Frontend Implementation

> Skip if the plan says "No frontend changes required."

1. Read the **Frontend Plan** section from `plan.md`.
2. Read the design reference (based on `CLAUDE.md` Design Reference mode):
   - `REPO_DIR`: Analyze the committed design files for visual guidance.
   - `FIGMA_MCP`: Fetch relevant frames via Figma MCP.
   - `URL`: Use the plan's design reference notes for guidance. Do not attempt to fetch the URL.
   - `NONE`: Implement a clean, professional UI following tech stack conventions.
3. Implement UI components per the plan:
   - Follow `coding_standards.md` Section 3 (Frontend Guidelines).
   - Add `data-testid` attributes to all interactive elements per Section 3.6.
   - Implement i18n if specified in the tech stack.
4. Commit: `feat({FEATURE_ID}): implement frontend components`

---

## 9. Phase B — Backend Implementation

> Skip if the plan says "No backend changes required."

1. Read the **Backend Plan** section from `plan.md`.
2. Implement per plan:
   - Router/controller with input validation
   - Service layer with business logic
   - Repository layer with data access
   - Database migrations (if any)
   - Follow `coding_standards.md` Section 2 (Backend Guidelines)
3. If this is the first backend feature and no `.env.example` exists yet, create one with all required environment variables (database URL, API keys as placeholders, etc.) with descriptive comments.
4. **Unit tests:** Generate and run immediately.
   - Write to the directory specified in `CLAUDE.md` → Test Configuration → Unit test directory.
   - Follow naming convention from `CLAUDE.md` → Test Configuration → Unit test naming.
   - Must cover: happy path, at least 1 edge case, at least 1 error case per service method.
   - Follow `testing_standards.md` Section 1.1.
   - **All tests must pass before committing.** If tests fail, fix the code and re-run (max 2 attempts). If still failing, log the failures and proceed — note them in the PR.
5. **Integration tests** (if `Integration Tests: ENABLED` in `CLAUDE.md`):
   - Write to the directory specified in `CLAUDE.md` → Test Configuration → Integration test directory.
   - Follow naming convention from `CLAUDE.md` → Test Configuration → Integration test naming.
   - Generate repository and router integration tests.
   - Follow `testing_standards.md` Section 1.2.
   - Run tests. Must pass before committing.
6. Commit: `feat({FEATURE_ID}): implement backend`

---

## 10. Phase C — Integration

> Skip if the plan says "No external API integration" AND there is no frontend-to-backend connection needed.

1. Read the **API Integration Plan** and **API Contract** sections from `plan.md`.
2. If the project has both frontend and backend:
   - Connect frontend API service to real backend endpoints.
   - Replace any mock data with real HTTP calls.
   - Ensure request/response shapes match the contract.
3. If the project consumes external APIs:
   - Implement the API client module per `coding_standards.md` Section 4.
   - Connect frontend components to the API client.
   - Handle all error cases (network, auth, rate limiting, etc.).
4. Commit: `feat({FEATURE_ID}): connect to APIs`

---

## 11. Phase D — E2E Test Generation

> Skip if `E2E Tests: DISABLED` in `CLAUDE.md`.

1. Read the **Testing Strategy** and **Acceptance Test Outline** from `plan.md`.
2. Generate E2E test specs from acceptance criteria:
   - Write to the directory specified in `CLAUDE.md` → Test Configuration → E2E → Test directory.
   - Follow file naming from `CLAUDE.md` → Test Configuration → E2E → File naming.
   - One or more test cases per acceptance criterion.
   - At least one edge-case spec (empty state, boundary input, error condition).
   - Follow `testing_standards.md` Section 1.3 (locator precedence, isolation, deterministic data).
3. Commit: `test({FEATURE_ID}): add E2E test specs`
4. **Do NOT run E2E tests.** They are executed by CI when the PR is opened.

---

## 12. Phase E — Self-Review

Run the self-review checklist from `.claude/rules/review_standards.md` through the read-only `reviewer` subagent (`.claude/agents/reviewer.md`). The reviewer inspects but does not edit; it returns findings.

1. **Architecture & isolation** — verify layer discipline, no cross-module imports.
2. **Security** — no hardcoded credentials, input validation, transactional boundaries.
3. **Scope containment** — every acceptance criterion met, no gold plating.
4. **Test coverage** — verify tests exist for all required layers.
5. **Code quality** — no TODOs, no console output, consistent naming.

If the reviewer reports defects, the `builder` subagent applies the fixes:
- Fix BLOCKING defects (max 2 self-fix cycles).
- Re-run affected tests after each fix.
- If defects persist after 2 cycles, document them in the PR description as known issues.

Commit any fixes: `fix({FEATURE_ID}): address self-review findings`

---

## 13. Phase F — Refactor Gate

> Skip if `Refactor Gate: DISABLED` in `CLAUDE.md`.

1. Read `.claude/rules/refactoring_standards.md`.
2. Analyze all files created or modified by this feature against the refactoring checklist.
3. Classify findings as `RECOMMENDED` or `OPTIONAL`.
4. Apply all `RECOMMENDED` changes automatically.
5. Re-run unit and integration tests to verify no regressions.
   - If tests fail: revert the refactoring commit and log it.
6. Commit: `refactor({FEATURE_ID}): code quality cleanup`
7. Persist report to `.claude/artifacts/{FEATURE_ID}/refactor_report.md`.

---

## 14. Phase G — UAT Generation

> Skip if `UAT Generation: DISABLED` in `CLAUDE.md`.

1. Read the **Acceptance Test Outline** from `plan.md`.
2. **Generate Gherkin scenarios:**
   - Write to the directory specified in `CLAUDE.md` → Test Configuration → UAT → Scenarios directory.
   - Follow file naming from `CLAUDE.md` → Test Configuration → UAT → Gherkin file naming.
   - One Feature block per feature. Feature line includes the feature ID.
   - Background for shared setup (login, navigation, data seeding).
   - One Scenario per acceptance criterion minimum.
   - At least one edge-case Scenario.
   - Reference `data-testid` attributes where applicable.
3. **Generate manual UAT script:**
   - Write to the directory specified in `CLAUDE.md` → Test Configuration → UAT → Scripts directory.
   - Follow file naming from `CLAUDE.md` → Test Configuration → UAT → Manual script naming.
   - Also write a copy to `.claude/artifacts/{FEATURE_ID}/uat_script.md`.
   - Structured markdown with prerequisites, numbered steps, expected results, pass/fail checkboxes, summary table.
4. Commit: `test({FEATURE_ID}): add UAT scenarios and manual script`

---

## 15. Documentation Check

Compare the current repo state against `README.md` and `docs/DEVELOPMENT.md`:

### README.md
- If top-level directories were **created or removed** by this feature (e.g., `backend/`, `frontend/`), update the repository structure section.
- If Docker files, run scripts, or dev-server configuration were **created or changed**, update the "Running the project" section with real prerequisites and commands (e.g., `docker compose up`, port numbers, environment setup).
- If `.env.example` was created, mention it in the README.

### docs/DEVELOPMENT.md
- If this is the scaffold feature and infrastructure was just created, update the "Development Environment" prerequisites section with concrete requirements (Docker version, Node.js version, Python version, etc.) based on what was actually set up.
- If test infrastructure was created, add a "Running tests locally" section with the actual commands.

If neither applies, skip.

Commit if updated: `chore({FEATURE_ID}): update documentation`

---

## 16. Push and Update PR

1. Push all commits to the feature branch: `git push origin {branch}` (the test gate hook runs here; if tests fail, fix and re-push).
2. Via Git provider MCP, update the PR:
   - **Update title** to: `feat({FEATURE_ID}): {title}` (replaces the plan-phase title).
   - **Convert from draft** to ready for review.
   - **Update the PR description** with:
     - Implementation summary (what was built)
     - Infrastructure scaffolded (if scaffold feature)
     - Test summary (which tests were generated, pass/fail results)
     - Any assumptions made about ambiguous requirements
     - Known issues from self-review (if any)
     - Acceptance criteria checklist (checked off)

---

## 17. Update Status → In Review

Update the feature's status in the tracker to the `in_review` mapped status.
