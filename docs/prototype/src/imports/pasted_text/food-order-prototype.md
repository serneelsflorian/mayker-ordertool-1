Build a fully clickable prototype for a group food-ordering web app. A single restaurant is preselected/hardcoded ("Trattoria Demo"). One admin sets up a menu, shares it, teammates add their meals and submit when done, the admin closes the order and exports a consolidated list to manually re-enter into Deliveroo. There is no real backend and no real link-sharing — simulate everything in-app with state.
Brand colors (use these as the palette; apply via the chosen design system): primary teal #269A91, accent red/coral #D44858, light blue-grey #9ABFCB, warm taupe #A39286. Use teal as the primary action color, coral for destructive/alert actions (close order, validation errors) and the prank-email accent, and the blue-grey and taupe as supporting/neutral tones for surfaces and secondary text. Ensure sufficient contrast for accessibility.
Demo navigation (important): This is a single app with a persistent role switcher in a top bar that toggles between Admin view and Guest view, so a reviewer can see both sides without real URLs. Both views read and write the same in-memory order state, so changes in one reflect in the other. Include a small "reset demo" control that restores the seeded starting state.
Seed the prototype with sample data so screens look populated on load: a menu of ~8 items (Margherita 9.50, Diavola 12.00, Carbonara 13.50, Caesar Salad 8.00, Tiramisu 6.50, Garlic Bread 4.00, Sparkling Water 3.00, House Red 5.50), each with a category (Pizza, Pasta, Salad, Dessert, Sides, Drinks). Pre-populate 3 guests with varied items, notes, and quantities — and varied submit status so the admin overview shows a mix: e.g. "Sara: 2x Margherita; 1x Tiramisu — Submitted", "Tom: 1x Diavola - extra spicy; 1x Garlic Bread — Editing", "Mira: 1x Caesar Salad - no croutons; 1x Sparkling Water — Submitted". The order starts in the open state.
Admin view — screens and behavior:

Menu setup: Shows the hardcoded restaurant name (not editable) and a form to add menu items: name (required text), price (required, positive number, 2 decimals), category (optional text). Adding with an empty name or a non-numeric/negative price shows an inline validation error (coral) and does not add the item. Added items appear immediately in a list below showing name, price, and category. Each item has a remove control that updates the list instantly.
Generate link: A "Generate share link" button disabled until at least one menu item exists. Activating it produces a fake share URL like app.example/order/abc123 in a read-only field with a one-tap Copy button showing a "Copied" confirmation. After generating, the order is "open."
Live order overview (admin only): A panel showing all guests' selections grouped by person, each with a per-guest status badge — "Editing" or "Submitted" — and a live grand total, updating as guest-view changes are made. Include a summary indicator at the top of this panel showing how many have submitted (e.g. "2 of 3 submitted"). Use teal for "Submitted" and a neutral tone (taupe/blue-grey) for "Editing."
Close order: A "Close order" button (coral) visible while open. Clicking it shows a confirmation dialog ("Close order? Members can no longer make changes"). Confirming sets state to closed. Closing is final — no reopen control. Once closed, guest controls are disabled everywhere.
Export (enabled only when closed): A "Generate export" action displaying plain, copy-paste-friendly text with a one-tap Copy all button. Export starts with the restaurant name as a header, then groups identical items as quantity x item name (e.g. 3x Margherita). Items merge only when both the item and its note match — different notes are separate lines with the note beneath (e.g. 1x Margherita - no onions). Ends with a final total equal to the sum of all selections.
Email the export (enabled only when closed): An "Email overview" button opening a modal with a recipient field (required, email-format validated) plus optional CC and BCC fields (validated as email format when non-empty). Invalid/empty recipient shows an inline error (coral) and blocks send. The modal shows a full email preview containing the consolidated export (restaurant header, grouped lines with notes, total) and a clearly tongue-in-cheek prank line: "Thank you for ordering! The bill will be sent to your email shortly, so keep an eye on your inbox 😉" — phrased to read as an obvious joke. A "Send" button shows a fake "Sent" confirmation toast; order state is unchanged.

Guest view — screens and behavior:

Shows the restaurant name and full menu as read-only, selectable items. No login or account step.
A required name field at the top; add controls are disabled until a non-empty name is entered.
The guest can add any menu item; each addition is tagged with their entered name. Each added item has an optional free-text note field and a quantity stepper (minimum 1). The guest can remove their own added items.
A running subtotal of only the current guest's own selections, updating on every change.
Submit flow (new): A prominent "Submit my order" button (teal). Submitting marks the guest as Submitted — this is a soft submit: their items are now flagged done and the admin sees the "Submitted" badge, but the guest can still click "Reopen / edit my order" to return to Editing status and change items. After submitting, show a clear confirmation state in the guest view (e.g. a banner: "Your order is submitted — the organizer can see it. You can still reopen to make changes."). When a submitted guest edits anything, either prompt them to reopen first or automatically flip their status back to "Editing" so the admin's view stays accurate. The submit button should be disabled if the guest has added no items.
The guest can only see, edit, or remove their own items — never other guests'.
When the order is closed: the menu is read-only, all add/edit/remove/quantity/submit controls are disabled, and a clear banner states "This order is closed — no more changes can be made."

Cross-cutting: Make every interaction work for real (validation, add/remove, quantity, totals, submit/reopen with status syncing to the admin overview, close, export text generation, email preview, copy confirmations). Include empty states (menu with no items, guest with nothing added, no one submitted yet) and disabled/closed states. Use clear confirmation toasts. Make it responsive and mobile-friendly, since guests will likely open it on phones.
